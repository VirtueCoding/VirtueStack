# VirtueStack Comprehensive Codebase Audit: Gaps, Risks, and Production Readiness

> **Date:** 2026-03-28
> **Scope:** Full repository audit — 84,597 lines of Go, 65 migrations, 130+ API endpoints, 2 frontend apps, 38 gRPC methods
> **Methodology:** Static analysis of every layer (API handlers, services, repositories, models, tasks, middleware, node agent, storage backends, frontend, migrations, tests, deployment)

---

## Executive Summary

VirtueStack is a **surprisingly mature** single-developer infrastructure project. The architecture is sound — clean layered design (Handler → Service → Repository → PostgreSQL), production-grade security (Argon2id, RLS, CSRF, timing-attack mitigation, mTLS), and genuine multi-backend storage support (Ceph RBD, QCOW2, LVM). The codebase has no TODO/FIXME/HACK markers and passes all 25 golangci-lint rules.

**However, there are critical gaps that would prevent production deployment for a paying customer base.** This document maps every gap found, categorized by severity, with specific file citations and remediation plans.

---

## Critical Gaps (Must Fix Before Production)

### 1. No VM State Machine — Status Transitions Are Unvalidated

**Files:** `internal/controller/models/vm.go` (8 status constants), `internal/controller/services/vm_service.go` (all UpdateStatus calls)

**The Problem:** VM states (`provisioning`, `running`, `stopped`, `suspended`, `migrating`, `reinstalling`, `error`, `deleted`) are defined as string constants, but **no transition validation exists**. Any code path can set any status from any state via `UpdateVMStatus()`, which is a bare `UPDATE vms SET status = $1 WHERE id = $2`.

**Impact:**
- A VM in `deleted` state can be set back to `running` by a concurrent task
- Two tasks racing on the same VM can both succeed, creating an inconsistent state
- A VM stuck in `provisioning` due to a crash has no automated recovery — it stays in `provisioning` forever
- Application-level checks (e.g., "check VM is stopped before starting") have a TOCTOU race window

**Fix Required:**
```sql
-- Database-enforced status values
ALTER TABLE vms ADD CONSTRAINT vms_status_check
  CHECK (status IN ('provisioning','running','stopped','suspended','migrating','reinstalling','error','deleted'));
```

```go
// Repository-level optimistic locking
func (r *VMRepository) TransitionStatus(ctx context.Context, vmID, fromStatus, toStatus string) error {
    result, err := r.db.Exec(ctx,
        "UPDATE vms SET status = $1, updated_at = NOW() WHERE id = $2 AND status = $3",
        toStatus, vmID, fromStatus)
    if result.RowsAffected() == 0 {
        return fmt.Errorf("transition %s→%s: VM not in expected state: %w", fromStatus, toStatus, sharederrors.ErrConflict)
    }
    return err
}
```

Plus a transition map that defines which transitions are legal:
```go
var validVMTransitions = map[string][]string{
    "provisioning":  {"running", "error"},
    "running":       {"stopped", "suspended", "migrating", "reinstalling", "error"},
    "stopped":       {"running", "deleted", "reinstalling", "migrating", "error"},
    "suspended":     {"running", "stopped", "deleted"},
    "migrating":     {"running", "error"},
    "reinstalling":  {"running", "error"},
    "error":         {"stopped", "deleted"},
}
```

**Severity:** 🔴 Critical — data corruption risk in production with concurrent operations

### 2. Incomplete VM Creation Cleanup on Failure

**File:** `internal/controller/tasks/handlers_vm_create.go`

**The Problem:** VM creation is a multi-step process (clone disk → generate cloud-init → allocate IPs → create VM → start VM). Cleanup on failure is **partial and inconsistent**:

| Step Failure | Cleanup Performed | Gap |
|---|---|---|
| `GenerateCloudInit` fails | ✅ Deletes cloned disk | — |
| `CreateVM` (gRPC) fails | ✅ Deletes disk + releases IPs | — |
| `StartVM` fails | ❌ **No cleanup** | Orphaned VM definition + disk + IPs remain |
| MAC address update fails | ❌ **No cleanup** | VM running but metadata inconsistent |
| Template ID update fails | ❌ **No cleanup** | VM running but metadata wrong |
| Status update to `running` fails | ❌ **No cleanup** | VM running but status stuck at `provisioning` |

**Impact:** Failed-to-start VMs leave orphaned resources (disk, IPs, libvirt domain) that consume capacity but are invisible to customers. Over time, this causes "phantom" resource exhaustion.

**Fix Required:** Implement a compensation stack pattern where each successful step registers its cleanup function, and failure at any point triggers rollback of all previous steps in reverse order. Cleanup errors should be logged (not silently discarded with `_ =`).

**Severity:** 🔴 Critical — resource leaks in production

### 3. No OpenAPI/Swagger Specification

**Files:** All of `internal/controller/api/admin/`, `customer/`, `provisioning/` — 130+ endpoints with zero machine-readable documentation

**The Problem:** Searching the entire codebase for "swagger", "openapi", "swag" returns zero results. There is no API specification that third-party integrators (WHMCS module developers, custom automation scripts, SDK generators) can use.

**Impact:**
- WHMCS module developers must read Go source code to understand API contracts
- No client SDK generation possible
- No automated contract testing between frontend and backend
- API breaking changes are invisible until runtime

**Fix Required:** Add `swaggo/swag` annotations to handler functions and generate OpenAPI 3.0 spec as part of the build.

**Severity:** 🟡 High — blocks third-party integration ecosystem

---

### 4. No Stuck-Task Recovery

**Files:** `internal/controller/tasks/worker.go`, `internal/controller/models/task.go`

**The Problem:** If the Controller crashes while a task is in `running` state, the task stays `running` forever. NATS JetStream's redelivery (3 attempts, 5-minute ack wait) helps for tasks that never ack'd, but tasks that were ack'd and then the process crashed remain stuck.

**Current mitigations:**
- Tasks have `started_at` timestamps — but nothing reads them to detect stuck tasks
- No periodic scanner to find tasks stuck in `running` beyond a reasonable timeout

**Fix Required:** Add a background goroutine that periodically scans for tasks where `status = 'running' AND started_at < NOW() - INTERVAL '30 minutes'` and resets them to `pending` for reprocessing (with retry count check).

**Severity:** 🟡 High — tasks silently never complete after Controller restart

---

### 5. No Rate Limiting on Password Reset

**Files:** `internal/controller/api/customer/auth_password_reset.go`, `internal/controller/api/customer/routes.go`

**Status from `features.md`:** Password reset endpoints exist and are implemented.

**The Problem:** While login endpoints have rate limiting (5 attempts/15 minutes), the password reset endpoint needs explicit rate limiting to prevent:
- Email enumeration (timing differences between existing/non-existing emails)
- Email bombing (triggering thousands of reset emails to a victim)

**Fix Required:** Add dedicated rate limiter: 3 requests/hour per email, 10 requests/hour per IP.

**Severity:** 🟡 High — abuse vector in production

## Important Gaps (Fix Before Scale)

### 6. God-File: server.go (852 Lines)

**File:** `internal/controller/server.go`

**The Problem:** Single file handling HTTP server setup, 15+ service initializations, all repository creation, background scheduler starts, health checks, request logging, and response formatting. Any change to service wiring requires reading through 852 lines.

**Fix Required:** Split into `server.go` (lifecycle), `dependencies.go` (DI wiring), `schedulers.go` (background tasks), `response.go` (helpers).

**Severity:** 🟡 Medium — maintainability degradation over time

---

### 7. God-Service: node_agent_client.go

**File:** `internal/controller/services/node_agent_client.go`

**The Problem:** Single file containing gRPC connection pooling, ALL VM lifecycle operations, ALL storage operations, ALL network operations, ALL backup operations, and ALL guest agent operations. This is the largest service file in the codebase.

**Fix Required:** Split by domain: `node_agent_vm.go`, `node_agent_storage.go`, `node_agent_network.go`, `node_agent_guest.go`.

**Severity:** 🟡 Medium — maintainability

---

### 8. Frontend Code Duplication

**Files:** `webui/admin/components/ui/` (~18 files), `webui/customer/components/ui/` (~19 files)

**The Problem:** Both frontend apps duplicate identical shadcn/ui component files, API client base infrastructure, auth provider patterns, and Tailwind configuration. Updating a Button component requires editing two files.

**Fix Required:** Extract shared packages into a monorepo structure (Turborepo or npm workspaces) with `packages/ui/`, `packages/api-client/`, `packages/config/`.

**Severity:** 🟡 Medium — double maintenance burden

---

### 9. No Proto Breaking-Change Detection

**Files:** `proto/virtuestack/node_agent.proto` (972 lines, 38 RPC methods)

**The Problem:** No CI check validates that proto changes are backward-compatible. A breaking change (renaming a field, changing a field number, removing an RPC) would only be caught at runtime when Controller tries to call an old Node Agent.

**Fix Required:** Add `buf` to CI with `buf breaking --against .git#branch=main`.

**Severity:** 🟡 Medium — silent breakage during upgrades

---

### 10. No Golden-File Tests for Domain XML

**File:** `internal/nodeagent/vm/domain_xml.go`

**The Problem:** Libvirt domain XML generation is a critical path — malformed XML silently breaks VM creation with cryptic libvirt errors. There are no snapshot tests comparing generated XML against known-good baselines.

**Fix Required:** Add golden-file tests that generate domain XML for representative configurations (Ceph disk, QCOW disk, LVM disk, with/without ISO, various NIC configs) and compare against checked-in reference files.

**Severity:** 🟡 Medium — hard-to-diagnose VM creation failures

---

### 11. Missing System-Level Webhook Events

**Files:** `internal/controller/services/webhook.go` — customer-scoped only

**The Problem:** Webhooks exist for customer events, but there are no system-level event notifications for:
- Node going offline/degraded
- Failover triggered
- Storage pool capacity warnings
- Admin actions (VM migration, plan changes)

Hosting providers need these for integration with their monitoring/alerting systems (PagerDuty, Slack, custom dashboards).

**Fix Required:** Extend webhook system with a system-level event bus. Events publish to NATS subjects that both internal handlers and external subscribers can consume.

**Severity:** 🟡 Medium — blocks operational monitoring integration

## Lower Priority Gaps (Nice to Have)

### 12. No Structured Logging in Task Handlers

**Files:** `internal/controller/tasks/handlers_vm_create.go`, `handlers_vm_delete.go`, etc.

**The Problem:** Task handlers use `slog` but don't consistently include structured fields like `task_id`, `vm_id`, `node_id` in every log line. Diagnosing a failed task requires correlating log entries manually.

**Fix Required:** Create a task-scoped logger helper that automatically includes task metadata in every log entry:
```go
logger := taskLogger(task) // includes task_id, task_type, vm_id from payload
logger.Info("cloning disk", "template", templateID)
```

**Severity:** 🟢 Low — operational convenience

---

### 13. No Health Check for NATS Connection

**Files:** `internal/controller/server.go` (healthHandler function)

**The Problem:** The health endpoint checks PostgreSQL connectivity but not NATS JetStream. If NATS goes down, the Controller appears healthy but cannot process any async tasks.

**Fix Required:** Add NATS health check to the readiness endpoint. Return `503` if NATS connection is not `CONNECTED`.

**Severity:** 🟢 Low — misleading health status

---

### 14. No Database Connection Pool Metrics

**Files:** `internal/controller/metrics/` — Prometheus metrics exist for HTTP requests but not DB pool

**The Problem:** pgx connection pool stats (active connections, idle connections, wait time, wait count) are not exported to Prometheus. Database connection exhaustion is a common production issue that's invisible without these metrics.

**Fix Required:** Add Prometheus collector for `pgxpool.Stat()`:
```go
prometheus.MustRegister(prometheus.NewGaugeFunc(
    prometheus.GaugeOpts{Name: "db_pool_total_conns"},
    func() float64 { return float64(pool.Stat().TotalConns()) },
))
```

**Severity:** 🟢 Low — observability improvement

---

### 15. Customer Self-Registration Not Implemented

**File:** `features.md` — Gap #3, marked as "deferred (not needed for WHMCS-only)"

**The Problem:** Customers can only be created via WHMCS provisioning API or admin panel. Direct self-registration with email verification doesn't exist. This limits VirtueStack to WHMCS-only deployments.

**Fix Required (when needed):** `POST /customer/auth/register` with email verification flow, configurable via `ALLOW_SELF_REGISTRATION=true` env var (default: disabled).

**Severity:** 🟢 Low — only needed for non-WHMCS deployments

## What's Already Working Well (Don't Touch)

These components are production-ready and should not be refactored:

- ✅ **Storage abstraction** — 3 backends (Ceph RBD, QCOW2, LVM) with clean interface, factory pattern, and domain-specific error types
- ✅ **Auth system** — Argon2id, JWT, 2FA, API keys, timing-attack mitigation, session management
- ✅ **RLS multi-tenancy** — PostgreSQL row-level security with `SetCustomerContext()` wrapper
- ✅ **Three-tier API** — Admin/Customer/Provisioning with distinct auth models and rate limits
- ✅ **Task queue** — NATS JetStream with durable streams, retry backoff, idempotency keys
- ✅ **Circuit breaker** — Production-grade 3-state model for failover operations
- ✅ **Heartbeat checker** — 15s interval, 3-miss threshold, non-invasive health monitoring
- ✅ **Cursor-based pagination** — Keyset pagination with (n+1) pattern, stable under concurrent inserts
- ✅ **RBAC** — 26 fine-grained permissions, 3 roles, helper functions
- ✅ **Backup/snapshot unification** — Clean single-concept model with `method` field (from features.md Gap #1)
- ✅ **WHMCS integration** — Customer auto-provisioning, SSO, usage metering all implemented
- ✅ **Template management** — Build from ISO, distribution to nodes, cache tracking
- ✅ **Console access** — VNC (noVNC) + Serial (xterm.js) with WebSocket proxy
- ✅ **Node failover** — STONITH + Ceph blocklist + VM migration orchestration
- ✅ **Audit system** — Sensitive field masking, request/response logging, per-resource audit trails

## Summary

**Total findings:** 15 gaps identified

| Severity | Count | Items |
|----------|-------|-------|
| 🔴 Critical | 2 | VM state machine (#1), creation cleanup (#2) |
| 🟡 High | 3 | OpenAPI (#3), stuck tasks (#4), password reset rate limiting (#5) |
| 🟡 Medium | 6 | God-files (#6, #7), frontend duplication (#8), proto checks (#9), golden tests (#10), system webhooks (#11) |
| 🟢 Low | 4 | Task logging (#12), NATS health (#13), DB metrics (#14), self-registration (#15) |

**Overall assessment:** VirtueStack is **80% production-ready**. The remaining 20% is concentrated in two critical gaps (state machine + cleanup) that represent real data corruption risk, and a handful of operational maturity items that can be addressed incrementally.

**The most impactful single change is implementing VM state machine validation** — it touches ~20 call sites across task handlers and services, but prevents an entire class of production bugs that would otherwise require manual database intervention to resolve.
