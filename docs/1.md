# VirtueStack Architectural Audit: What I'd Do Differently Starting Over

> **Date:** 2026-03-28
> **Context:** Full knowledge of the current codebase (84,597 lines Go, 65 migrations, 38 gRPC methods, 12 async task types, 130+ API endpoints). This is NOT a refactor wishlist. This is "if I cleared the codebase and wrote it again, what decisions change and why."

---

## Open-Core vs. Fully Open-Source: The Monetization Question

Before the 12-layer analysis — yes, **open-core is the right model** for VirtueStack. Here's the detailed comparison:

### Fully Open-Source (AGPL/MIT/Apache)

| Pros | Cons |
|------|------|
| Maximum community adoption | Revenue limited to support contracts, hosting, or donations |
| Contributors fix bugs for free | Competitors fork and sell without contributing back (MIT/Apache) |
| Trust signal for infrastructure software | No funding for sustained solo development |
| Proxmox model proves it can work at scale | Proxmox has 20+ years and enterprise support contracts — different game |

### Open-Core (Open base + proprietary premium)

| Pros | Cons |
|------|------|
| Core platform free = low adoption barrier | "Open-core" perceived negatively by some OSS purists |
| Premium features fund continued development | Must clearly define the open/proprietary boundary |
| Hosting providers **expect** to pay for production tooling | Community may be smaller if premium features are what they need |
| Aligns incentives: better product = more revenue | Maintaining two codebases (open + proprietary) adds overhead |

### Recommendation: Open-Core with Clear Boundary

**Open (AGPL-3.0):** Core VM management, single-node operation, customer portal, REST API, basic templates, backup/snapshot, QCOW2 storage backend

**Proprietary tier:** HA clustering/failover orchestration, multi-node live migration, WHMCS billing integration, Ceph RBD integration, white-label branding, advanced analytics dashboards, priority support, LVM thin provisioning

**Why AGPL specifically:** Prevents SaaS competitors from running a modified fork without contributing back. MIT/Apache would let AWS wrap VirtueStack and sell it as a managed service. AGPL closes that loophole while keeping the core genuinely open.

**Why this boundary:** The open tier must be genuinely useful on its own — a single-node VPS host with QCOW2 storage is a real product. The proprietary tier targets scale operators who need HA, Ceph, and billing integration — and who have revenue to pay for it.

## 1. LANGUAGE AND FRAMEWORK CHOICE

**Current approach:** Go 1.26 with Gin web framework (`go.mod`), pgx/v5 for PostgreSQL, NATS JetStream for task queue, gRPC+protobuf for Controller↔Node Agent, Next.js 16 + React 19 for two separate frontend apps (`webui/admin/package.json`, `webui/customer/package.json`).

**Verdict: KEEP** ✅

This is one of the strongest decisions in the codebase. Every choice is defensible:

- **Go for infrastructure software is correct.** The cloud-native ecosystem (Docker, Kubernetes, Terraform, Prometheus) is Go. VirtueStack's target operators already run Go tooling. Single-binary deployment, zero runtime dependencies, excellent concurrency primitives, and fast compilation are exactly what a VM management platform needs. Rust would offer memory safety guarantees but dramatically increase development time for a solo developer. Python/Node would sacrifice the single-binary deployment model and add runtime dependency management overhead.

- **Gin is adequate.** It's the most popular Go HTTP framework. For this project size, the standard library `net/http` with chi router would reduce dependency surface — but Gin works and switching gains nothing meaningful. The middleware chain pattern in `internal/controller/api/middleware/` (auth, CSRF, rate limiting, audit, correlation IDs) maps naturally to Gin's handler chain.

- **pgx/v5 over an ORM is correct.** The repository layer (`internal/controller/repository/`) uses direct SQL with parameterized queries. For a project with RLS (`SET app.current_customer_id`), partitioned audit logs, cursor-based pagination (`repository/cursor/pagination.go`), and complex filter queries, an ORM like GORM would fight you constantly. The custom scanner pattern (e.g., `ScanRow[T]()`, `ScanRows[T]()` generics in `repository/db.go`) is verbose but explicit and type-safe.

- **NATS JetStream** for the task queue is lighter than RabbitMQ/Kafka, embeddable for testing, and the durable stream with retry policies (`tasks/worker.go`: 3 attempts, exponential backoff 10s→60s→300s, 7-day retention) is well-configured.

- **gRPC for Node Agent communication is correct.** Bidirectional streaming for VNC/serial console (`StreamVNCConsole`, `StreamSerialConsole` in `node_agent.proto`), mTLS requirement, and the proto-defined contract make gRPC the obvious choice over REST for internal service communication.

**What I'd tweak (not change):**
- Consider **chi** router instead of Gin for `net/http` compatibility — but this is a preference, not a deficiency.

**Effort impact:** Keeping Go+Gin saves years of rewrite time. The language choice is paying dividends already.

**Migration note:** No migration needed. The stack is sound.

## 2. PROJECT STRUCTURE AND MODULE BOUNDARIES

**Current approach:** Monorepo with `internal/controller/`, `internal/nodeagent/`, `internal/shared/`. Controller has `api/` (3 tiers: admin, customer, provisioning), `services/`, `repository/`, `models/`, `tasks/`. Clean layered architecture: Handler → Service → Repository → PostgreSQL.

**Verdict: CHANGE** (targeted, not wholesale)

**What's good:**
- No circular dependencies — the package graph is clean. Build succeeds without import cycle issues.
- Three-tier API separation (`admin/`, `customer/`, `provisioning/`) correctly models distinct authorization domains with different auth (JWT+2FA, JWT-or-APIKey, APIKey-only), rate limits (500/min, 100 read+30 write/min, 1000/min), and scopes.
- `internal/shared/` for cross-cutting concerns (errors, config, crypto, logging) is well-factored.
- The `DB` interface in `repository/db.go` abstracting `*pgxpool.Pool` and `pgx.Tx` is excellent design.
- Repository generic scanners (`ScanRow[T]()`, `ScanRows[T]()`) reduce boilerplate without an ORM.

**What I'd change:**

### Problem 1: `server.go` is an 852-line god-file

`internal/controller/server.go` handles HTTP server setup, 15+ service initializations, all repository creation, background scheduler start, health checks, and response formatting helpers. This is the single largest coupling point in the codebase.

**Fix:** Split into:
- `server.go` — HTTP server lifecycle only (Start/Stop/health)
- `dependencies.go` — Service and repository wiring (the DI container)
- `schedulers.go` — Background task schedulers (backup, heartbeat, metrics)
- `response.go` — Response formatting helpers (`respondJSON`, `respondError`, `ParsePagination`)

### Problem 2: Several god-services

The services directory (`internal/controller/services/`) has 40 files, but size distribution is highly uneven:
- `node_agent_client.go` — gRPC connection pooling, VM lifecycle ops, network ops, storage ops, backup ops, guest agent ops ALL in one file
- `backup_service.go` — backup creation, restoration, scheduling, Ceph-specific logic, QCOW-specific logic, LVM-specific logic, retry handling
- `vm_service.go` — VM creation, power operations, metrics fetching, node selection, failover preparation

Each should be split by responsibility:
- `node_agent_client.go` → `node_agent_vm.go`, `node_agent_storage.go`, `node_agent_network.go`, `node_agent_guest.go`
- `backup_service.go` → `backup_create_service.go`, `backup_restore_service.go`
- `vm_service.go` → `vm_lifecycle_service.go`, `vm_power_service.go`

### Problem 3: Handler duplication across API tiers

Admin, customer, and provisioning handlers for listing/getting VMs share near-identical pagination parsing, response formatting, and error handling — differing only in authorization scope. This should be extracted into shared handler helpers in a common package.

**Trade-off:** The current approach does work. Splitting files is cosmetic if you're the only developer. But when debugging a production issue at 3 AM, finding the relevant code in a 1500-line file vs. a 200-line file matters.

**Effort impact:** File-level reorganization within same packages — no import paths change. Can be done incrementally.

**Migration note:** Start with `server.go` split (highest impact). The rest can happen during normal maintenance.

## 3. DATABASE DESIGN AND ORM STRATEGY

**Current approach:** PostgreSQL 16 with 65 sequential migrations, pgx/v5 direct SQL, RLS for tenant isolation via `SET app.current_customer_id` session variable (`repository/db.go`), custom scanner functions per model, cursor-based keyset pagination (`repository/cursor/pagination.go`).

**Verdict: KEEP** ✅ (with one addition)

**What's excellent:**
- **PostgreSQL is the right database.** RLS for multi-tenant isolation is a killer feature. The `SetCustomerContext()` pattern in `repository/db.go` gives defense-in-depth that no application-layer-only approach matches. Even with a bug in application code, PostgreSQL prevents cross-tenant data access.
- **No ORM is the right call.** With RLS session variables, partitioned audit logs, cursor-based pagination, and 65 complex migrations, an ORM would constantly fight you.
- **Cursor-based pagination** in `repository/cursor/pagination.go` (151 lines) using the (n+1) pattern to detect final pages is excellent — avoids expensive `COUNT(*)` on large tables and provides stable pagination under concurrent inserts.
- **65 migrations with expand-contract pattern** shows discipline. Lock timeout enforcement prevents long-running DDL from blocking production traffic.

**What I'd add:**

**Missing: A query builder for dynamic filters.** Repository methods build filter queries by concatenating SQL fragments with conditional `WHERE` clauses. This is error-prone and grows unwieldy. I'd introduce **squirrel** (`github.com/Masterminds/squirrel`) for composing parameterized queries safely. It's not an ORM — it's a query compositor that produces raw SQL for pgx:

```go
// Before (current - manual string building):
query := "SELECT " + vmColumns + " FROM vms WHERE 1=1"
if filter.CustomerID != nil { query += " AND customer_id = $" + nextParam }
if filter.Status != nil { query += " AND status = $" + nextParam }

// After (with squirrel):
q := sq.Select(vmColumns...).From("vms")
if filter.CustomerID != nil { q = q.Where(sq.Eq{"customer_id": *filter.CustomerID}) }
if filter.Status != nil { q = q.Where(sq.Eq{"status": *filter.Status}) }
sql, args, _ := q.PlaceholderFormat(sq.Dollar).ToSql()
```

**Effort impact:** Additive — existing queries continue working. New queries use the builder. Gradual adoption.

**Migration note:** Install squirrel, use for new repository methods, optionally migrate existing complex filters.

## 4. API DESIGN

**Current approach:** REST with three tiers (`/api/v1/admin/*`, `/api/v1/customer/*`, `/api/v1/provisioning/*`). JSON request/response. JWT + API key auth. Standardized error format with correlation IDs (`middleware/recovery.go`). 130+ endpoints across admin (28 handler files), customer (31 files), provisioning (19 files).

**Verdict: KEEP** ✅

**What's correct:**
- **REST is the right choice.** API consumers are WHMCS (PHP), customer scripts, and two internal SPAs. GraphQL adds complexity for simple CRUD patterns. gRPC for the public API would alienate hosting providers who want `curl`-able endpoints.
- **Three-tier separation is excellent.** Admin, Customer, and Provisioning have genuinely different auth models, rate limits, and authorization scopes. Merging them would create permission matrix complexity in every handler.
- **API versioning via `/api/v1/`** is correctly implemented and future-proof.
- **Standardized error format** with `code`, `message`, and `correlation_id` is production-ready.
- **Dual-auth middleware** (`JWTOrCustomerAPIKeyAuth` in `middleware/auth.go`) elegantly handles browser sessions and programmatic access without duplicating routes.
- **Customer API key scoping** with 7 permission types (`vm:read`, `vm:write`, `vm:power`, `backup:read`, `backup:write`, `snapshot:read`, `snapshot:write`) and optional `vm_ids` restriction + `allowed_ips` CIDR whitelist is well-designed.

**What I'd improve:**
- **Add OpenAPI/Swagger spec generation.** There's currently NO machine-readable API spec (confirmed by searching for "swagger", "openapi" across the codebase). With 130+ endpoints, this is a significant gap. I'd add `swaggo/swag` annotations to auto-generate OpenAPI 3.0 specs — enabling client SDK generation, interactive docs, and contract testing.
- **Standardize on cursor-based pagination everywhere.** The codebase has cursor pagination infrastructure (`repository/cursor/pagination.go`) but some endpoints still use offset-based pagination. Standardize on cursor-based for all list endpoints.

**Effort impact:** OpenAPI annotations are additive (~1 comment block per handler). Pagination standardization is incremental.

**Migration note:** Add `swaggo/swag` and annotate new endpoints first. Backfill existing endpoints over time.

## 5. HYPERVISOR AND PROVIDER ABSTRACTION

**Current approach:** `StorageBackend` interface in `internal/nodeagent/storage/interface.go` (13+ methods) with three implementations: Ceph RBD (`rbd.go`), QCOW2 (`qcow.go`), LVM (`lvm.go`). Factory pattern in `factory.go` returns a `BackendPair{Storage, Template, Type}`. Separate `TemplateBackend` interface (5 methods). VM lifecycle via libvirt in `internal/nodeagent/vm/lifecycle.go`. Domain XML generation in `vm/domain_xml.go`.

**Verdict: KEEP** ✅

This is one of the best-designed abstractions in the codebase:

- **`StorageBackend` interface** with 13 methods covering the full lifecycle (clone, resize, snapshot, rollback, delete, pool stats) and three real implementations is clean and extensible. Adding a fourth backend (e.g., ZFS) is straightforward: implement the interface, add a case to the factory.
- **`TemplateBackend` interface** correctly separates template operations from general storage.
- **Error types are domain-specific** — `StorageErrorCode` enum (NOT_FOUND, QUOTA_EXCEEDED, IN_USE, NO_SPACE, etc.) rather than generic errors.
- **Factory pattern** in `storage/factory.go` creates both storage and template managers together, ensuring consistency.

**What I'd explicitly NOT change:**
- **No hypervisor abstraction layer.** The codebase is hardcoded to KVM/QEMU via libvirt. This is **correct for the product**. VirtueStack targets VPS hosting providers running bare-metal KVM. Adding a hypervisor abstraction for VMware/Xen/Hyper-V would add massive complexity for a use case that doesn't exist in the target market. libvirt is already the right abstraction level for KVM.
- **Domain XML generation** (`vm/domain_xml.go`) is correctly kept separate from lifecycle management.

**Effort impact:** No structural changes needed. The abstraction is clean and has already proven extensible (3 backends).

**Migration note:** N/A.

## 6. AUTHENTICATION, AUTHORIZATION, AND MULTI-TENANCY

**Current approach:** JWT with HttpOnly cookies + refresh tokens (`services/auth_service.go`). Argon2id password hashing (64MB/3iter/4parallel). TOTP 2FA with 10 backup codes (`services/auth_service_2fa.go`). Customer API keys with HMAC-SHA256 and 7 permission scopes. Provisioning API keys (SHA-256 hashed). RLS at database level (`repository/db.go`). RBAC with 3 admin roles + 26 fine-grained permissions (`models/permission.go`). Session limits (3 admin, 10 customer). Failed login lockout (5 attempts/15min).

**Verdict: KEEP** ✅

This is the most thoroughly-implemented layer in the codebase. Every decision is defensible:

- **Argon2id over bcrypt** is correct — memory-hard, resists GPU/ASIC attacks. Parameters follow OWASP recommendations.
- **Timing-attack mitigation** (`auth_service.go`): dummy hash computed at startup for non-existent users prevents email enumeration via response timing. Most projects miss this.
- **Session limits** with oldest-session eviction is production-appropriate.
- **Destructive action re-authentication** requiring fresh auth for delete/force-stop/migrate operations is a strong security pattern (enforced via `X-Reauth-Token` header).
- **Customer API key scoping** with `vm_ids` restriction and `allowed_ips` CIDR whitelist provides the right granularity.
- **RLS as defense-in-depth** — even with application bugs, PostgreSQL RLS prevents cross-tenant data access. The `CustomerScopedDB` wrapper in `repository/db.go` automatically sets the session variable for every customer-scoped query.
- **26 fine-grained permissions** across 3 roles (SuperAdmin, Admin, Viewer) with `HasPermission()`, `HasAnyPermission()`, `HasAllPermissions()` helpers is well-structured.

**What I'd add (not change):**
- **OAuth2/OIDC provider support** for customer login (Google/GitHub). This doesn't replace JWT — it adds an identity provider that issues JWTs after OIDC verification. Feature addition, not architectural change.

**Effort impact:** Auth layer is mature and hardened. Don't touch what works.

**Migration note:** OAuth2/OIDC is additive alongside existing email/password flow.

## 7. TASK QUEUE AND ASYNC OPERATIONS

**Current approach:** NATS JetStream durable stream ("TASKS") with 4 worker goroutines (`tasks/worker.go`). 3 retry attempts with exponential backoff (10s→60s→300s). 5-minute ack timeout. 12 task types registered via map-based handler dispatch (`tasks/handlers.go`). Task state tracked in PostgreSQL with `IdempotencyKey` deduplication. Progress tracking (0-100%). Circuit breaker for failover operations (`services/circuit_breaker.go`, 377 lines, 3-state model: Closed→Open→Half-Open).

**Verdict: CHANGE** (the infrastructure is right, but there's a critical gap in VM state management)

**What's good:**
- NATS JetStream is the right choice — lighter than Kafka, durable, with proper consumer groups
- Exponential backoff (10s→60s→300s) with 3 retries is well-configured
- Idempotency keys exist in the task table for deduplication — `vm.create`, `backup.create`, etc. all support idempotent retries
- Progress tracking (0-100%) enables UI progress bars
- Circuit breaker (`services/circuit_breaker.go`) with configurable thresholds prevents cascading failures during failover
- Task state lifecycle is clean: `SetRunning()`, `SetCompleted()`, `SetFailed()`, `SetProgress()` with injectable clock for testing

**What's critically missing — VM state machine:**

**There is NO formal state machine for VM lifecycle transitions.** VM states are defined as 8 string constants in `models/vm.go`, but there's **no transition validation**. Any code path can set any status from any state via repository `UpdateStatus()`. I verified this by examining `services/vm_service.go` — status changes are simple `UPDATE vms SET status = $1 WHERE id = $2` with no validation of legal transitions.

This means:
1. A VM in `provisioning` state could be set to `running` without completing provisioning
2. A VM in `deleted` state could be set back to `running`
3. Two concurrent tasks could race on state transitions without conflict detection
4. `StartVM()` checks the VM is `stopped` before starting — but this check is in application code, not enforced at the database level, creating a TOCTOU race window

**What I'd do differently:** Implement a state machine with validated transitions at the repository level:

```go
var validVMTransitions = map[string][]string{
    VMStatusProvisioning:  {VMStatusRunning, VMStatusError},
    VMStatusRunning:       {VMStatusStopped, VMStatusSuspended, VMStatusMigrating, VMStatusReinstalling, VMStatusError},
    VMStatusStopped:       {VMStatusRunning, VMStatusDeleted, VMStatusReinstalling, VMStatusMigrating, VMStatusError},
    VMStatusSuspended:     {VMStatusRunning, VMStatusStopped, VMStatusDeleted},
    VMStatusMigrating:     {VMStatusRunning, VMStatusError},
    VMStatusReinstalling:  {VMStatusRunning, VMStatusError},
    VMStatusError:         {VMStatusStopped, VMStatusDeleted},
}

func (r *VMRepository) TransitionStatus(ctx context.Context, vmID, fromStatus, toStatus string) error {
    // Validates transition AND uses WHERE status = fromStatus for optimistic locking
    result, err := r.db.Exec(ctx,
        "UPDATE vms SET status = $1, updated_at = NOW() WHERE id = $2 AND status = $3",
        toStatus, vmID, fromStatus)
    if result.RowsAffected() == 0 {
        return fmt.Errorf("state transition %s→%s failed: %w", fromStatus, toStatus, ErrConflict)
    }
    return err
}
```

This gives you optimistic locking (concurrent state changes don't silently overwrite), transition validation (illegal transitions caught at write time), and auditability.

**Effort impact:** This is the **highest-value change in this entire analysis**. It prevents an entire class of production bugs (ghost VMs in impossible states, data corruption from concurrent operations).

**Migration note:** Add `TransitionStatus()` alongside existing `UpdateStatus()`. Migrate callers one at a time. Add database CHECK constraint for valid status values. Run audit query to find VMs in impossible states.

## 8. STATE MANAGEMENT AND ERROR HANDLING

**Current approach:** Error types in `internal/shared/errors/errors.go` — 12 sentinel errors + `ValidationError` (field-level) + `OperationError` (multi-step with operation/step context) + `APIError` (HTTP response format). `RespondWithError()` in middleware for standardized HTTP errors. Partial failure cleanup in task handlers (manual try-catch pattern). Idempotency via `IdempotencyKey` column in tasks table. Deletion handlers are idempotent (succeed if resource doesn't exist).

**Verdict: KEEP** (error types are excellent; improve cleanup pattern)

**What's excellent:**
- **Error type hierarchy is well-designed.** Sentinel errors with `errors.Is()` support, typed errors with `Unwrap()` for chain traversal, `APIError` for HTTP layer translation. The `OperationError` type with `Operation` and `Step` fields is perfect for diagnosing multi-step failures.
- **Correlation IDs** (`middleware/correlation.go`) thread through the entire request lifecycle.
- **Idempotency is comprehensive** — task deduplication, deletion idempotency, webhook delivery tracking with idempotency keys.
- **VM deletion** (`tasks/handlers_vm_delete.go`) is well-implemented: graceful degradation on partial failures (non-critical cleanup steps logged but continue), idempotent (already-deleted VM = success).

**What I'd improve:**

The cleanup pattern in `tasks/handlers_vm_create.go` is **manual and incomplete**:

```go
// Current pattern (from handlers_vm_create.go):
// If GenerateCloudInit fails → delete cloned disk ✅
// If CreateVM fails → delete disk AND release IPs ✅
// If StartVM fails → NO cleanup ❌ (partially created VM left orphaned)
// If MAC/template update fails → NO cleanup ❌
```

I'd introduce a **compensation stack** pattern:

```go
type CleanupStack struct {
    steps []func() error
}
func (cs *CleanupStack) Push(fn func() error) { cs.steps = append(cs.steps, fn) }
func (cs *CleanupStack) Rollback(logger *slog.Logger) {
    for i := len(cs.steps) - 1; i >= 0; i-- {
        if err := cs.steps[i](); err != nil {
            logger.Error("cleanup step failed", "step", i, "error", err)
        }
    }
}
```

This ensures cleanup always runs in reverse order and **logs failures instead of silently discarding them** (the current pattern uses `_ =` to discard cleanup errors).

**Effort impact:** Low — a single helper struct that task handlers adopt incrementally.

**Migration note:** Add `CleanupStack` utility, use in new task handlers, backfill existing handlers during maintenance.

## 9. CONFIGURATION AND DEPLOYMENT

**Current approach:** Environment variables (primary) with YAML override (`internal/shared/config/config.go`). `Secret` type that redacts in logs/JSON via `String()` returning `[REDACTED]`. Docker Compose with 6 services (PostgreSQL, NATS, Controller, Admin UI, Customer UI, Nginx) on bridge network with resource limits and health checks. Non-root containers with `cap_drop: [ALL]` + selective `cap_add`. Node Agent runs as host binary (not containerized). AES-256 encryption key validated as exactly 32 bytes.

**Verdict: KEEP** ✅

**What's correct:**
- **Env vars as primary config** is correct for Docker/container deployments. YAML override exists for complex configurations.
- **`Secret` type** prevents accidental exposure in logs — a security feature many projects miss.
- **Docker Compose** configuration is production-grade: resource limits, health checks with start periods, `no-new-privileges`, `cap_drop: [ALL]`, non-root UIDs, named volumes.
- **Node Agent as host binary** is architecturally correct — it needs direct libvirt/KVM access. Containerizing would require `--privileged`, negating all container security benefits.
- **Database pool config** (25 max, 5 min connections, 1-hour lifetime) and **server timeouts** (10s read, 30s write, 120s idle) are production-appropriate.
- **Graceful shutdown** with 30-second grace period catches SIGINT/SIGTERM.

**What I'd add:**
- **`.env` validation script.** With 50+ env vars (some required: `DATABASE_URL`, `JWT_SECRET`, `ENCRYPTION_KEY`, `NATS_AUTH_TOKEN`), a pre-start validation that checks required variables exist and have valid formats would prevent the #1 support issue: "my stack won't start."
- **Helm chart** for Kubernetes deployments (hosting providers increasingly run their management plane on K8s).

**Effort impact:** Minimal for env validation. Helm chart is a nice-to-have.

**Migration note:** Add `scripts/validate-env.sh` that checks required variables.

## 10. FRONTEND ARCHITECTURE

**Current approach:** Two separate Next.js 16 apps (`webui/admin/`, `webui/customer/`) with React 19. shadcn/ui (Radix primitives + Tailwind CSS 3.4) for components. TanStack React Query 5.64 for server state. react-hook-form 7.71 + Zod 3.24 for forms. Centralized API client in `lib/api-client.ts` per app. Customer portal additionally uses: xterm 6.0 for serial console, noVNC 1.5 for graphical console, Recharts 3.8 for resource charts.

**Verdict: CHANGE** (consolidate shared infrastructure, keep separate deployable apps)

**What's good:**
- **Next.js 16 + React 19** is current and well-chosen. SSR, file-based routing, React concurrent features.
- **shadcn/ui** is the right component library — copy-paste-own rather than dependency-locked. Radix accessibility with full customization. Perfect for a product that needs to look professional without a design team.
- **TanStack Query** for server state eliminates an entire category of state management bugs.
- **No Zustand** — and that's fine. Auth context + TanStack Query covers everything needed. No complex client-side state exists that needs a global store.
- **Centralized API client** with CSRF handling, error typing, and refresh token management is well-structured.
- **Console integrations** (xterm + noVNC) are production-standard choices.

**What I'd change:**

**Problem: Duplicated infrastructure across two apps.**

Both `webui/admin/` and `webui/customer/` duplicate:
- Radix UI component files (`components/ui/` — 18-19 identical files)
- API client base infrastructure (fetch wrapper, error handling, CSRF)
- Auth provider pattern
- Tailwind configuration
- Package.json dependencies (identical versions of React, Next.js, TanStack Query)

**Fix:** Use a **monorepo with shared packages** (Turborepo or npm workspaces):

```
webui/
├── packages/
│   ├── ui/              ← Shared shadcn/ui components
│   ├── api-client/      ← Shared fetch wrapper, error types, CSRF
│   └── config/          ← Shared Tailwind/TS config
├── apps/
│   ├── admin/           ← Admin-specific pages + components
│   └── customer/        ← Customer-specific pages + components
├── package.json         ← Workspace root
└── turbo.json           ← Build pipeline config
```

**Trade-off:** Adds monorepo tooling complexity. For a solo developer, the overhead is non-trivial. But maintaining two codebases where updating a Button component requires editing two files is worse long-term.

**Effort impact:** Medium — restructuring requires moving files and updating imports, no logic changes.

**Migration note:** Start by extracting `components/ui/` into a shared package (highest duplication area). Then extract the API client base. Leave app-specific components where they are.

## 11. TESTING STRATEGY

**Current approach:** 91 Go test files. Table-driven tests with testify/assert+require. Mock interfaces with function fields (not mockery/gomock). Integration tests via Testcontainers (`tests/integration/` — 7 test files using PostgreSQL 16 + NATS). E2E tests via Playwright (`tests/e2e/` — 15+ spec files with page objects). `golangci-lint` with 25 linters. Race detector in CI (`make test-race`). Security tests directory exists (`tests/security/`).

**Verdict: CHANGE** (good foundation, critical gaps in coverage strategy)

**What's good:**
- **Table-driven tests with testify** is the correct Go testing pattern
- **Function-field mocks** are simpler than code-generated mocks for this project size and avoid mockery/gomock dependency
- **Testcontainers** for integration tests gives real PostgreSQL + NATS without Docker Compose ceremony
- **Playwright E2E** with page objects, multi-browser testing, and mobile viewports is comprehensive
- **Race detector** in CI catches concurrency bugs before production
- **25 linters** in golangci-lint (with `new: true` for incremental adoption) is well-configured

**What's missing:**

### Gap 1: No contract tests between Controller and Node Agent

The Controller calls Node Agent via gRPC (38 RPC methods in proto), but there's no test validating the proto contract between them. If someone changes the proto and regenerates only one side, the mismatch is caught only at runtime. Add **proto-breaking-change detection** via `buf` (`buf breaking --against .git#branch=main`).

### Gap 2: No golden-file tests for domain XML generation

`internal/nodeagent/vm/domain_xml.go` generates libvirt domain XML. Malformed XML silently breaks VM creation. Golden-file tests comparing generated XML against known-good snapshots would catch regressions immediately.

### Gap 3: Service-layer coverage gaps for critical paths

The 3 largest services (vm_service, backup_service, node_agent_client — thousands of lines combined) need more focused unit tests for edge cases: node selection with no available nodes, backup fail mid-stream, gRPC connection drop during VM creation.

### Gap 4: Load testing is minimal

`tests/load/` exists but is thin. For a platform managing potentially hundreds of concurrent API calls, load testing the Controller API (task creation = NATS publish + PostgreSQL write) is critical for production readiness.

**What I'd design from day one:**

```
Testing Pyramid:
├── Unit tests (services, models, utilities) — 70%
│   ├── Table-driven with mocked dependencies
│   ├── Edge cases: invalid states, concurrent access, resource exhaustion
│   └── Golden-file tests for domain XML generation
├── Integration tests (repository + real PostgreSQL) — 20%
│   ├── Testcontainers for DB + NATS
│   ├── RLS isolation verification
│   └── Migration up+down for every migration
├── Contract tests (Controller ↔ Node Agent) — 5%
│   ├── buf breaking for proto compatibility
│   └── gRPC mock server validation
└── E2E tests (Playwright) — 5%
    ├── Critical user flows only (login, VM create, backup)
    └── Smoke tests, not comprehensive coverage
```

**Effort impact:** Contract tests and golden-file tests are high-value, low-effort. Load tests are medium-effort but critical.

**Migration note:** Add `buf` to CI. Add golden-file tests for domain XML. Expand load tests for top 5 endpoints.

## 12. PLUGIN AND EXTENSION SYSTEM

**Current approach:** No formal plugin/extension system exists. Extension points are implicit via Go interfaces (StorageBackend, TemplateBackend, TaskPublisher, NodeAgentClient, IPAllocator, BandwidthThrottler, etc.). Task handlers registered via map-based dispatch. Middleware chain extensible via `router.Use()`. Webhook system exists for customer event callbacks.

**Verdict: RETHINK** (but don't build a plugin framework — build webhook-driven extensibility)

**What's correct about the current approach:**
- **Interface-based extensibility** is the right pattern for compile-time extension (adding a new storage backend, swapping an IP allocator). These are used by VirtueStack developers, not third parties.
- **10+ internal interfaces** provide clean seams for testing and alternative implementations.

**Why NOT to build a traditional plugin system:**
- Go's `plugin.Open()` is Linux-only, requires exact Go version match, and is fragile. It's effectively abandoned by the Go team.
- Infrastructure software doesn't need plugins the way a CMS does. Hosting providers want to customize billing, notifications, and customer workflows — not the hypervisor layer.
- A plugin SDK is enormous maintenance burden for a solo developer.

**What I'd build instead: System-level webhooks + event bus**

The webhook system already exists for customer event callbacks (`services/webhook.go` — HMAC-SHA256 signed async delivery). But it's customer-scoped only. I'd extend it to be the primary extensibility mechanism:

1. **System-level webhooks** (not just customer webhooks) that fire on every significant event: `vm.created`, `vm.deleted`, `backup.completed`, `node.offline`, `customer.created`, `payment.received`
2. **Pre-action hooks** (optional): `vm.creating` fires before VM creation, allowing an external system to approve/deny/modify the request (e.g., fraud check, custom billing validation, quota enforcement)
3. **NATS event subjects**: External services can subscribe to VirtueStack events via NATS without HTTP overhead — the infrastructure already exists (JetStream is running)

This gives third parties full extensibility without any plugin framework:
- Custom billing? Subscribe to `vm.created` and `vm.deleted` webhooks
- Custom monitoring? Subscribe to `node.heartbeat` events via NATS
- Approval workflows? Register a pre-action hook on `vm.creating`

**Trade-off:** Loses the ability for plugins to modify internal behavior (e.g., custom node selection algorithms). But the interface-based extension points handle those cases for developers willing to fork or contribute.

**Effort impact:** Medium — webhook infrastructure exists. Extending to system-level events and pre-action hooks is incremental.

**Migration note:** Start with system-level webhooks for top 10 events. Add NATS event subjects. Add pre-action hooks later based on demand.
