# Repo-Wide Audit and Remediation Design Spec

**Date:** 2026-04-01
**Scope:** Entire repository, including Go backend, Node Agent, frontends, PHP billing modules, CI/workflows, Docker artifacts, scripts, and documentation directly tied to fixes
**Goal:** Run a broad, evidence-driven audit across the codebase, fix confirmed issues immediately, and iterate until the selected audit methods and repository validation stop surfacing actionable findings

---

## Problem Statement

The requested work is broader than a normal bugfix: it combines security review, threat modeling, standards compliance, dead-code review, and opportunistic correctness fixes across a large multi-language repository. That creates two risks:

1. **Coverage risk** — a single pass through one part of the tree will miss issues in other domains
2. **Change risk** — repo-wide edits can damage unrelated work, especially because the current worktree already contains uncommitted changes and generated frontend build artifacts

The design must therefore maximize coverage without turning into speculative churn, and it must preserve unrelated local changes.

---

## Recommended Approach

Use a **parallel multi-agent audit with centralized triage and verification**.

The main session acts as the coordinator. Multiple background agents, all using `gpt-5.4`, audit separate domains in parallel and report only concrete issues. The main session then:

- deduplicates findings
- spot-checks high-risk claims
- fixes issues in small batches
- reruns the relevant repository-native validation
- repeats the audit/fix loop until no new confirmed findings appear from the completed audit methods

This approach balances breadth and safety better than a single sequential review or a security-only pass.

### Why this approach

- **Broader coverage:** each domain gets focused attention
- **Lower false-positive rate:** the main session only fixes verified findings
- **Safer changes:** targeted fix batches reduce blast radius
- **Measurable progress:** each batch closes a concrete finding and is verified immediately

### Alternatives considered

#### 1. Verification-first single stream

Run tests, builds, and linters first, then investigate only failing areas.

**Pros:** low risk, reproducible, efficient
**Cons:** weak at finding latent security issues, dead code, and standards violations that do not fail CI

#### 2. Threat-model-first security review

Start with auth, SSRF, secrets, webhooks, CI supply chain, and external integrations, then expand to correctness and cleanup.

**Pros:** strong security depth
**Cons:** narrower total coverage; non-security correctness and maintainability issues are easier to miss

The recommended design keeps the verification discipline of the first alternative while retaining the coverage benefits of the parallel audit.

## Operating Constraints

### Existing dirty worktree

The repository already contains unrelated uncommitted changes, including generated `webui/admin/.next/*` output and source/documentation edits. The audit must therefore:

- work on top of the current tree
- avoid reverting or overwriting unrelated changes
- avoid touching generated artifacts unless they are regenerated by an intentional validation step
- keep edits surgical and confined to confirmed findings

### Definition of “no issues found”

The work cannot prove the absolute absence of bugs or vulnerabilities. The practical success condition is:

> no remaining confirmed findings from the audit methods completed in this session, plus no newly introduced failures in the repository-native validation relevant to touched areas

That means the audit is evidence-driven, not claim-driven.

### Model and agent policy

All sub-agents will use `gpt-5.4`, which is the strongest shared model setting exposed in this environment. A separate `xhigh` flag is not available via the tool interface, so the design standardizes on `gpt-5.4` explicitly.

## Audit Architecture

### Coordinator role

The main session is responsible for:

- launching audit agents in parallel
- maintaining the canonical issue list
- validating whether findings are real, duplicated, or out of scope
- applying fixes
- running validation commands after each fix batch
- deciding whether another audit round is needed

### Parallel audit domains

The audit is split into independent streams:

1. **Go backend and shared packages**
   - handlers, services, repositories, shared utilities
   - error handling, input validation, auth, concurrency, correctness

2. **Node Agent and infrastructure-facing Go code**
   - libvirt/Ceph logic, gRPC handlers, lifecycle flows, storage and networking logic
   - privilege boundaries, external command use, resource cleanup, safety checks

3. **Frontend TypeScript/React**
   - admin and customer UIs
   - type safety, unsafe API handling, dead code, standards drift, broken UX paths

4. **PHP billing modules**
   - WHMCS and Blesta modules
   - request validation, webhook safety, error handling, secret handling

5. **CI, Docker, scripts, and supply chain**
   - GitHub Actions pinning and workflow safety
   - Dockerfile hardening, image practices, script robustness, env handling

6. **Cross-cutting standards and dead-code review**
   - violations of `docs/coding-standard.md`
   - dead code, unreachable code, unused helpers, stale docs directly coupled to touched logic

### Agent output contract

Each audit agent must return only findings that meet all of these conditions:

- points to a specific file and code location or clearly scoped artifact
- explains the concrete issue and impact
- states why it violates correctness, security, or `docs/coding-standard.md`
- includes a suggested verification method
- avoids speculative or style-only feedback

This keeps the remediation queue actionable.

## Triage and Remediation Workflow

### Finding triage

Every reported issue is classified as one of:

- **confirmed-fix-now** — concrete, in scope, safe to remediate immediately
- **confirmed-but-defer** — real, but too invasive for the current batch and must be explicitly queued
- **not-a-finding** — speculative, duplicate, already addressed, or unsupported

The default bias is to fix real issues, but not to churn code based on weak evidence.

### Fix batching

Fixes are applied in small coherent batches, usually one domain or one tightly related issue set at a time. Each batch should:

- preserve existing architecture patterns
- reuse existing helpers where possible
- avoid unrelated refactors
- include documentation updates only when directly coupled to the fix

### Re-audit loop

After each fix batch:

1. rerun the relevant validation
2. if the fix changed behavior significantly, request an additional review pass
3. continue with remaining queued findings
4. if the queue is empty, run one final targeted audit sweep to look for newly exposed issues

## Verification Strategy

Only existing repository commands and checks will be used.

### Baseline checks

Before making code changes, run baseline validation to understand the current state:

- Go: `make test`
- Frontend: workspace lint/type-check/build commands already used by CI
- PHP: syntax checks for module files
- targeted repo checks that correspond to touched files

Pre-existing failures found at baseline must be recorded and not misattributed to subsequent fix batches.

Where native library constraints make certain full checks impractical, use the non-native baseline that is already supported by the repository and call out any resulting verification gap explicitly in the final summary.

### Post-fix validation matrix

| Area touched | Validation |
|---|---|
| Go controller/shared code | `make test`, plus targeted `go test` where useful |
| Node Agent-adjacent code | if native headers are available, run the relevant existing build/test path; otherwise use the strongest non-native check available and record the limited verification scope |
| Admin frontend | existing lint, type-check, and build commands |
| Customer frontend | existing lint, type-check, and build commands |
| PHP modules | `php -l` on touched files or module tree |
| Workflows / scripts / Docker | targeted syntax or build validation already present in repo tooling |

### Completion bar

The task is considered complete only when:

- all confirmed findings selected for remediation are fixed
- relevant validations pass for touched areas
- a final audit pass does not surface additional confirmed issues that can be fixed safely in-session

## Threat Modeling Focus Areas

The audit should give extra scrutiny to:

- authentication and authorization boundaries
- multi-tenant isolation and RLS assumptions
- SSRF and URL fetch paths
- webhook verification and external callbacks
- secret handling in code, scripts, CI, and Docker
- unsafe process execution and shell usage
- race conditions, stale state transitions, and multi-step task failures
- generated token handling, session lifecycle, and refresh flows
- supply-chain exposure in workflows and image builds

These are the highest-value areas for a repo-wide security pass.

---

## Non-Goals

The audit is not intended to:

- perform broad style cleanup with no correctness or security value
- replace working architecture purely for elegance
- rewrite generated artifacts by hand
- introduce new tooling that the repository does not already use
- revert unrelated user changes in the dirty worktree

---

## Expected Deliverables

1. A verified issue list with duplicates and weak claims removed
2. Concrete code and config fixes for confirmed findings
3. Targeted documentation updates where directly required
4. Validation evidence for each fix batch
5. A final summary of what was fixed, what was checked, and any remaining explicitly deferred items
